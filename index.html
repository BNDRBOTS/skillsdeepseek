⚠ BREAKING: Claude Opus 4.7 Literal Instruction Following

Opus 4.7 interprets prompts more literally than Opus 4.6. Soft language like "consider," "you might," or "feel free to" is now interpreted as optional. Directives like "you must" or "always" are interpreted as required. The model will not silently generalize an instruction from one item to another, and will not infer requests you didn't make. Sampling parameters (temperature, top_p, top_k) return HTTP 400 if set to non-default values. Old budget_tokens-style extended thinking payloads also return 400.

2.2 OpenAI Custom GPT Instructions & GPT-5.5 API

Custom GPT Instructions Field

Plain text box in the Configure tab. Use Markdown headings and trigger/instruction pairs: Trigger: [condition] Instruction: [action]. Frame instructions positively. Maximum safe limit: ~32,000 characters.

GPT-5.5 Official 7-Part Prompt Structure

Role: [1-2 sentences defining function, context, and job]
# Personality [tone, demeanor, and collaboration style]
# Goal [user-visible outcome]
# Success criteria [what must be true before the final answer]
# Constraints [policy, safety, business, evidence, and side-effect limits]
# Output [sections, length, and tone]
# Stop rules [when to retry, fallback, abstain, ask, or stop]

Critical GPT-5.5 Prompting Rules

  1. Start with the smallest prompt that preserves the product contract — then tune reasoning effort, verbosity, tool descriptions, and output format.
  2. Avoid carrying over every instruction from an older prompt stack. Legacy prompts over-specify the process and can add noise, narrow the model's search space, or lead to overly mechanical answers.
  3. Personality and Collaboration style are two separate concerns: "Personality controls how the assistant sounds... Collaboration style controls how the assistant works." Keep both short. Neither should replace clear goals, success criteria, tool rules, or stopping conditions.
  4. Keep each section short. Add detail only where it changes behavior.
  5. Outcome-first: "GPT-5.5 is strongest when the prompt defines the target outcome, success criteria, constraints, and available context, then lets the model choose the path. For many tasks, describe the destination rather than every step."

⚠ JSON MODE — CRITICAL REQUIREMENT

To use JSON output: (1) Set response_format to {'type': 'json_object'}. (2) The system or user prompt MUST contain the word "json" and include an example JSON format. (3) Set max_tokens appropriately to prevent JSON truncation.

From the docs: "The incoming system or user prompt MUST contain the word 'json' and provide a sample of the desired JSON output format so the model can produce legal JSON."

System Prompt Best Practices

  • System prompt: 80–150 characters, verb-first. DeepSeek has low system-prompt sensitivity; avoid nested conditions, filler words, and content over 80 characters. User instructions have higher priority. Keep system prompts short and verb-first.
  • Use verb-first directives. Avoid nested conditions, filler words, and content exceeding ~80 characters.
  • System prompt counts toward total token consumption. Excessive length squeezes user conversation space and increases cost.
  • User instructions have higher priority than system instructions in DeepSeek's architecture — avoid conflicts.

2.4 Gemini System Instruction

⚠ GEMINI 3.1 PRO — SILENT TRUNCATION

The system_instruction field has a limit of 2048 Unicode characters. Content exceeding this length is silently truncated without error. If you pass 3000 characters, the model only sees the first 2048. The rest are discarded with no warning. This is the single most consequential constraint for Gemini skills.

Gemini uses system_instruction as an independent field — not merged into the user message flow but participating in attention weight initialization as an independent context anchor. This improves role consistency and instruction compliance rate.

PART 4 — CROSS-PLATFORM STRUCTURAL PRIMITIVES

4.1 XML Tags — Claude's #1 Structural Tool

Anthropic's official documentation explicitly states XML tags are "the single most impactful structural tool available inside Claude prompts." Claude has a native affinity for XML — it was trained on vast amounts of XML-structured data and treats tags as strong attention anchors. Best practices: use consistent descriptive tag names, nest tags for natural hierarchies, and wrap different content types in dedicated tags (<instructions>, <context>, <input>).

💡 CROSS-MODEL NOTE

XML tags are optimized for Claude. For multi-model use, consider YAML in fenced code blocks or Markdown section headers instead.

4.2 Markdown Headings — OpenAI's & Universal Fallback

OpenAI's official Custom GPT instructions guide explicitly recommends Markdown syntax with # headings. For multi-model prompts, Markdown headings serve as the universal structural delimiter recognized by all four platforms.

4.3 The Convergence — What Actually Works Everywhere

  1. Labeled sections — XML for Claude, Markdown # for OpenAI/GPT-5.5, prefixes for Gemini. Universal fallback: Markdown headings.
  2. Role/persona definition at the top — OpenAI puts "Role" first in its 7-part structure; Anthropic states "Give Claude a role… even a single sentence makes a difference."
  3. Constraints as discrete, verifiable rules — One per line, independently checkable by the model.
  4. Output format specification — All platforms support structured output specification.

PART 6 — BREAKING DEFAULTS & CREATIVE CONTROL

6.1 The Claude Opus 4.7 Default Aesthetic Problem

Claude Opus 4.7 has stronger design instincts with a consistent default house style: warm cream/off-white backgrounds (~#F4F1EA), serif display type (Georgia, Fraunces, Playfair), italic word-accents, and a terracotta/amber accent. This default is persistent. Generic instructions ("don't use cream," "make it clean and minimal") tend to shift to a different fixed palette rather than producing variety.

Two approaches work reliably per Anthropic's official documentation:

  1. Specify a concrete alternative with explicit color palettes, typefaces, and spacing rules.
  2. Have the model propose options before building: "Before building, propose 4 distinct visual directions tailored to this brief (each as: bg hex / accent hex / typeface — one-line rationale)."

6.2 The frontend-design Skill — 42 Lines That Changed Everything

Anthropic's official frontend-design skill is 42 lines long and produces dramatically better UI than generic prompts. The Medium analysis by Ahmed Soliman identified these key techniques:

  1. Name the failure mode: "This skill guides creation of distinctive, production-grade frontend interfaces that avoid generic 'AI slop' aesthetics." The phrase "AI slop" explicitly tells the model: the thing you are most likely to produce is exactly what I don't want.
  2. Force a design direction commitment before code: "Pick an extreme: brutally minimal, maximalist chaos, retro-futuristic, organic/natural, luxury/refined, playful/toy-like, editorial/magazine, brutalist/raw, art deco/geometric, soft/pastel, industrial/utilitarian." This is a chain-of-thought trigger disguised as design vocabulary.
  3. Ban specific defaults by name: "NEVER use generic AI-generated aesthetics like overused font families (Inter, Roboto, Arial, system fonts), cliched color schemes (particularly purple gradients on white backgrounds), predictable layouts and component patterns."
  4. Anticipate and block escape hatches: "NEVER converge on common choices (Space Grotesk, for example) across generations." The skill author noticed that Space Grotesk had become a new AI default and pre-emptively blocked it.
  5. Bind implementation to intent: "Match implementation complexity to the aesthetic vision. Maximalist designs need elaborate code with extensive animations and effects. Minimalist or refined designs need restraint, precision, and careful attention to spacing."
  6. End with permission: "Remember: Claude is capable of extraordinary creative work. Don't hold back, show what can truly be created when thinking outside the box and committing fully to a distinctive vision." Functions like a temperature-raise without actually changing the sampling parameter.

💡 THE VERIFIED ANTI-DEFAULT FORMULA

1. Explicit palette override (specify hex values — the only thing that consistently overrides Claude's cream default)
2. Anti-pattern specification (list exact defaults to avoid by name)
3. Concrete aesthetic reference (name specific design movements: "brutalist," "editorial," "art deco")
4. Option-then-build workflow (propose multiple directions first, then implement the chosen one)

6.3 GPT-5.5 Personality Block

OpenAI's official guide provides explicit personality configuration. Two dimensions:

  • Personality: controls tone, warmth, directness, formality — how the assistant sounds.
  • Collaboration style: controls when to ask questions, when to act autonomously, level of proactivity — how the assistant works.

Task-focused example: "You are a capable collaborator: approachable, steady, and direct. Assume the user is competent and acting in good faith, and respond with patience, respect, and practical helpfulness. Prefer making progress over stopping for clarification when the request is already clear enough to attempt."

Expressive variant: "Adopt a vivid conversational presence: intelligent, curious, playful when appropriate, and attentive to the user's thinking."

PART 7 — TOKEN EFFICIENCY: WHAT RESEARCH ACTUALLY SHOWS

7.1 Verified Findings

The claim that "prompt length is inversely correlated with instruction compliance beyond ~500 tokens" with specific compliance percentages does not appear in any verified academic research. What we actually know:

  1. System prompt length has significant drawbacks: Research demonstrated a 3000× reduction in prompt length while retaining ~98% of downstream performance. Long system prompts increase inference latency, computational cost, and reduce effective context length.
  2. Context length exhibits non-linear effects: One study found optimal performance around 8K–16K tokens for certain tasks.
  3. Prompt design significantly impacts compliance: "LLM compliance and accuracy are highly prompt-dependent… minor prompt changes can cause large changes in the distribution of generated labels."
  4. OpenAI's official position: "Start with the smallest prompt that preserves the product contract, then tune reasoning effort, verbosity, tool descriptions, and output format." "Keep each section short. Add detail only where it changes behavior."
  5. DeepSeek's official position: "Keep system instructions within a clear and concise range, avoiding excessive length that occupies the context window."

7.2 The Actual Principle

The verified principle is not a specific token count but a structural approach: discrete, labeled, verifiable constraints outperform long prose paragraphs because each constraint is independently checkable by the model. This is consistent with OpenAI's trigger/instruction pair approach, Anthropic's XML-tagged section approach, and Gemini's prefix-based demarcation approach.

11.2 Building Your Personal Anti-Default Vocabulary

The frontend-design skill taught us the critical insight: after you ban the obvious defaults (Inter, purple-on-white, rounded cards), watch what the model retreats to. Then ban those too. The second-order defaults (Space Grotesk, specific indie fonts) are where AI slop hides. Build your list incrementally:

  1. Run 5 design requests without anti-pattern bans. Note every recurring pattern.
  2. Ban those patterns by name in your next 5 requests.
  3. Note the new recurring patterns that emerge.
  4. Repeat until the output is genuinely unpredictable while remaining high-quality.

PART 12 — INSIDER TIPS MOST DEVELOPERS MISS

12.1 Claude Opus 4.7 Migration Traps

  1. Don't trust old prompts. "If your prompts used to work on 4.6, test them fresh on 4.7. The literal instruction following means soft language like 'consider' is now truly optional."
  2. Remove all sampling parameters. temperature, top_p, top_k — any non-default value returns 400. Use prompt-based alternatives for deterministic behavior: "be precise and consistent" instead of temperature=0.
  3. Use effort instead of thinking budgets. Old budget_tokens returns 400. Migrate to thinking={"type": "adaptive", "effort": "xhigh"}.
  4. Opt into visible thinking. Thinking text is hidden by default. If your app streams reasoning to users, explicitly request it: thinking={"type": "adaptive", "effort": "high", "display": "summarized"}.
  5. Measure token deltas. The new tokenizer maps same input to 1.0–1.35× more tokens. Measure before assuming bills stay flat.

12.2 GPT-5.5 Power Moves

  1. The greeting mechanism. Before any tool calls for multi-step tasks, send a short user-visible update acknowledging the request. Reduces perceived latency.
  2. Phase parameter. Use commentary for intermediate updates and final_answer for terminal output. If replaying messages, preserve original phase tags — otherwise the model may treat intermediate output as final.
  3. Personality ≠ Collaboration style. Keeping them separate prevents "model behavior confusion" — personality shapes user experience, collaboration shapes task execution. Mixing them causes unpredictable outputs.
  4. Build from scratch. OpenAI explicitly warns: "To get the most out of GPT-5.5, treat it as a new model family to tune for, not a drop-in replacement. Begin migration with a fresh baseline."

12.3 DeepSeek V4 Efficiency Secrets

  1. System prompt: 80–150 characters, verb-first. DeepSeek has low system-prompt sensitivity; avoid nested conditions, filler words, and content over 80 characters. User instructions take priority — keep system prompts short and verb-first.
  2. JSON mode requires the literal word "json" in the prompt. Without it, response_format: json_object won't work reliably. Always include an example JSON structure.
  3. Two length limits: max_tokens (API layer) and maxIterTimes (internal decoder iteration limit). Even with high max_tokens, output truncates if maxIterTimes hits first.

12.4 Gemini 3.1 Pro Silent Killers

  1. 2048-character silent truncation on system_instruction. Always verify with API response usage fields that actual token consumption matches expectation.
  2. Temperature >1.5 causes semantic fracture. About 15% of responses show topic jumps or mid-sentence grammar shifts. Keep creative work at 0.85–1.2.
  3. MoE architecture causes output fluctuation. Use structured prompts, post-processing, or application-layer caching to mitigate variance.
  4. Context window middle information decays. Place important information at the beginning or end of prompts.
  5. Safety settings are strict by default. Technical terms may be incorrectly blocked. Adjust thresholds for your use case.

12.5 Skill Discovery — Getting Your Skills Used

  1. The description field IS your trigger. Claude uses the description exclusively to decide when to invoke a Skill. Write it as a routing rule: "Use this skill when the user asks to build web components, pages, or applications."
  2. Single responsibility per Skill. "Monolithic instructions that try to do everything → Split into 2–3 single-purpose Skills and chain them."
  3. Progressive disclosure. Keep SKILL.md focused. Put long guides or examples in references/ or assets/. The model loads the full SKILL.md once activated.
  4. Eval-first development. Define 3–5 representative tasks and negative-trigger cases before expanding features. Gate releases on eval pass/fail.
