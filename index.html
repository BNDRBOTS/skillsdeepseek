PART 2: PLATFORM-SPECIFIC SKILL/PROMPT FORMATS — EXACT SPECIFICATIONS

2.1 Anthropic Claude — Custom Skills (claude.ai & Claude Code)

File Structure

Every Skill consists of a directory containing at minimum a Skill.md file, which is the core of the Skill. This file must start with a YAML frontmatter.

my-skill/
├── SKILL.md          # Required: YAML frontmatter + Markdown body
├── scripts/          # Optional: executable code
├── references/       # Optional: supplemental documentation
└── assets/           # Optional: templates, resources

YAML Frontmatter — Exact Field Limits

FieldRequiredLimitPurpose
nameYes64 characters maxHuman-friendly display name
descriptionYes200 characters max"This is critical—Claude uses this to determine when to invoke your Skill"
dependenciesNoFree texte.g. python>=3.8, pandas>=1.5.0

Source: Official Claude Help Center.

The metadata serves as the first level of a progressive disclosure system, providing just enough information for Claude to know when the Skill should be used without loading all content. The Markdown body is the second level — Claude accesses it after reading metadata.

Packaging

Zip the Skill folder. The ZIP must contain the Skill folder as its root (not files directly in ZIP root).

Trigger Mechanism

Claude uses the description field exclusively to decide when to invoke the Skill. Best practices from the official docs: "Write clear descriptions: Claude uses descriptions to decide when to invoke your Skill. Be specific about when it applies."

What Breaks Claude Skills

  • name > 64 characters → Skill fails to register
  • description > 200 characters → truncated, Claude may not trigger correctly
  • Code execution must be enabled in Customize → Skills
  • Skills can build on each other: "Claude can use multiple Skills together automatically."

2.2 OpenAI — Custom GPT Instructions & API System Prompt

Custom GPT Instructions Field

Plain text box in the Configure tab. Key guidelines from OpenAI's official help center:

  • Use Markdown headings (#, ##) and structured formatting for readability.
  • Use trigger/instruction pairs separated by delimiters: Trigger: [condition] Instruction: [action]
  • Avoid negative instructions: "Frame instructions positively to improve adherence and avoid confusion."
  • Break down steps granularly: "especially when multiple actions are required within a single step."
  • Promote attention to detail: Use "take your time," "take a deep breath," and "check your work" techniques.

GPT-5.5 API — The Official Prompt Structure

OpenAI's official prompt guidance for GPT-5.5 provides this suggested prompt structure for complex prompts:

Role: [1-2 sentences defining the model's function, context, and job]
# Personality [tone, demeanor, and collaboration style]
# Goal [user-visible outcome]
# Success criteria [what must be true before the final answer]
# Constraints [policy, safety, business, evidence, and side-effect limits]
# Output [sections, length, and tone]
# Stop rules [when to retry, fallback, abstain, ask, or stop]

Source: OpenAI API Docs — Prompt Guidance page.

The core philosophy: "GPT-5.5 is strongest when the prompt defines the target outcome, success criteria, constraints, and available context, then lets the model choose the path. For many tasks, describe the destination rather than every step."

Critical GPT-5.5 Prompting Rules

  1. "Start with the smallest prompt that preserves the product contract" — then tune reasoning effort, verbosity, tool descriptions, and output format against representative examples.
  2. "Avoid carrying over every instruction from an older prompt stack. Legacy prompts often over-specify the process because earlier models needed more help staying on track. With GPT-5.5, that can add noise, narrow the model's search space, or lead to overly mechanical answers."
  3. Personality and Collaboration style are two separate concerns: "Personality controls how the assistant sounds... Collaboration style controls how the assistant works." Both should be kept short.
  4. "Keep each section short. Add detail only where it changes behavior."

GPT-5.4 Specific Guidance

GPT-5.4 "benefits from more explicit guidance" in specific patterns: low-context tool routing, dependency-aware workflows, reasoning effort selection, research tasks, irreversible actions, and terminal/coding-agent environments. "Start with the smallest prompt that passes your evals, and add blocks only when they fix a measured failure mode."

JSON Mode — Critical Requirement

To use JSON output with DeepSeek, you must:

  1. Set response_format to {'type': 'json_object'}
  2. The system or user prompt MUST contain the word "json" and include an example JSON format to guide the model to output legal JSON
  3. Set max_tokens appropriately to prevent JSON truncation
用户传入的 system 或 user prompt 中必须含有 json 字样，并给出希望模型输出的 JSON 格式的样例

System Prompt Best Practices

  • Keep system instructions "clear and concise, avoiding excessive length that occupies context window."
  • System prompt counts toward total token consumption. "Excessively long system_prompt squeezes user conversation space and increases cost."
  • Avoid conflicts between system prompt and user input instructions.

2.4 Google Gemini — System Instruction

API Configuration

Gemini uses the system_instruction parameter as an independent field in the request — it is not merged into the user message flow but participates in attention weight initialization as an independent context anchor. This improves role consistency and instruction compliance rate.

from google.genai import types
response = client.models.generate_content(
    model="gemini-3.1-pro-preview",
    contents="...",
    config=types.GenerateContentConfig(
        system_instruction="You are a cat. Your name is Neko."
    )
)

Source: Gemini Cookbook.

CRITICAL CONSTRAINT — Gemini 3.1 Pro System Instruction Length
The system_instruction field has a limit of 2048 Unicode characters. Content exceeding this length is silently truncated without error. This is the single most consequential constraint for Gemini skills.

System Instruction Best Practices

  • System instruction defines the model's persona and global constraints, persisting across the entire conversation.
  • "Best Prompt combination: System Instruction sets the framework, each request's Prompt only writes the specific task — do not repeat constraint conditions. This maximizes token utilization."
  • Use prefixes (e.g., "English:", "JSON:") as semantic markers to help the model parse input and structure output.

PART 4: CROSS-PLATFORM STRUCTURAL PRIMITIVES — WHAT ACTUALLY WORKS

4.1 XML Tags — Anthropic's #1 Structural Recommendation

Anthropic's official documentation explicitly states: "XML tags help Claude parse complex prompts unambiguously, especially when your prompt mixes instructions, context, examples, and variable inputs. Wrapping each type of content in its own tag (e.g. <instructions>, <context>, <input>) reduces misinterpretation."

Best practices from the official docs:

  • Use consistent, descriptive tag names across prompts.
  • Nest tags when content has a natural hierarchy (documents inside <documents>, each inside <document index="n">).
  • Claude Opus 4.7's own system prompt uses 9 XML tags wrapped in a root <claude_behavior> tag.

Cross-model portability note: XML tags are optimized for Claude. For multi-model use, consider YAML in fenced code blocks or Markdown section headers instead.

4.2 Markdown Headings — OpenAI's Recommended Structure

OpenAI's official Custom GPT instructions guide explicitly recommends Markdown syntax: "Use the number sign # followed by a space to create headings... More number signs indicate smaller heading levels." They demonstrate # Context, # Instructions, # Additional Information as the canonical example.

4.3 Prefixes/Semantic Markers — Gemini's Preferred Approach

Google's Gemini cookbook recommends prefixes: "Prefixes are words or phrases followed by a colon used to demarcate different parts of a prompt, helping the model parse information effectively."

4.4 The Convergence: What Actually Works Everywhere

Based on verified documentation, the only structural elements that function reliably across all four platforms are:

  1. Labeled sections using whichever delimiter the target model favors (XML for Claude, Markdown # for OpenAI, prefixes for Gemini). The universal fallback is Markdown headings with clear labels.
  2. Role/persona definition at the top — OpenAI explicitly puts "Role" first in its 7-part structure; Anthropic states "Give Claude a role... even a single sentence makes a difference"; Gemini uses system_instruction for persona definition.
  3. Constraints as discrete, verifiable rules — OpenAI's "Constraints" section, Anthropic's "be clear and direct," DeepSeek's system prompt for behavioral rules.
  4. Output format specification — All platforms support structured output specification.

PART 5: PRODUCTION HARDENING — PROMPT CACHING, IDEMPOTENCY, DETERMINISM

5.1 Prompt Caching — Available on All Four Platforms

All four providers offer prompt caching:

  • Anthropic: Explicit cache markers. Cached input at ~10% of standard cost.
  • OpenAI: Automatic caching for GPT-5.5. "If the first N tokens of your prompt match a recent request, you pay 90% less for those tokens."
  • Google: Automatic caching (implicit).
  • DeepSeek: Automatic caching (implicit). KV cache isolation via user_id parameter.

Production implication: The system prompt (or Skill.md frontmatter + initial instructions) forms the stable prefix that gets cached. Make the prefix identical across requests for cache hits. This means: write the role, task, and constraints once, identically, and put variable data last.

PART 6: THE "EDGY TURN-UP" — VERIFIED MECHANISMS FOR BREAKING DEFAULTS

6.1 The Claude Opus 4.7 Default Aesthetic Problem

Claude Opus 4.7 has "stronger design instincts" with a consistent default house style: "warm cream/off-white backgrounds (~#F4F1EA), serif display type (Georgia, Fraunces, Playfair), italic word-accents, and a terracotta/amber accent." This default "is persistent. Generic instructions ('don't use cream,' 'make it clean and minimal') tend to shift the model to a different fixed palette rather than producing variety."

Two approaches work reliably per Anthropic's official documentation:

  1. Specify a concrete alternative with explicit color palettes, typefaces, spacing rules.
  2. Have the model propose options before building: "Before building, propose 4 distinct visual directions tailored to this brief (each as: bg hex / accent hex / typeface — one-line rationale)."

Additionally, Claude Opus 4.7 "requires less frontend design prompting than previous models to avoid generic patterns that users call the 'AI slop' aesthetic."

6.2 OpenAI GPT-5.5 Personality Block

The official GPT-5.5 guide provides explicit personality configuration blocks. A task-focused assistant example:

# Personality
You are a capable collaborator: approachable, steady, and direct.
Assume the user is competent and acting in good faith, and respond
with patience, respect, and practical helpfulness. Prefer making
progress over stopping for clarification when the request is already
clear enough to attempt.

An expressive variant:

# Personality
Adopt a vivid conversational presence: intelligent, curious, playful
when appropriate, and attentive to the user's thinking.

Source: OpenAI Prompt Guidance.

6.3 The Verified Anti-Default Formula

Based on the verified documentation across platforms, the mechanism that reliably breaks models out of default design patterns is:

  1. Explicit palette override (Claude: specify hex values directly; this is the only thing that consistently overrides its cream default)
  2. Anti-pattern specification (list exact defaults to avoid — OpenAI recommends this for output contracts)
  3. Concrete aesthetic reference (name specific design movements, eras, or reference works)
  4. Option-then-build workflow (have the model propose multiple directions first, then implement the chosen one — this is Anthropic's own recommendation for breaking Claude's design defaults)

PART 7: TOKEN EFFICIENCY — WHAT THE RESEARCH ACTUALLY SHOWS

7.1 What We Actually Know

The claim that "prompt length is inversely correlated with instruction compliance beyond ~500 tokens" with specific compliance percentages does not appear in any verified academic research. What we actually know:

  1. System prompt length has significant drawbacks: "Carefully engineered system prompts play a critical role in guiding the behavior of LLM agents, but their considerable length introduces significant drawbacks, including increased inference latency, higher computational cost, and reduced effective context length." Research demonstrated a 3000x reduction in prompt length while retaining ~98% of downstream performance.
  2. Context length exhibits non-linear effects: One study found "optimal performance around 8K-16K tokens" for certain tasks.
  3. Prompt design significantly impacts compliance: "LLM compliance and accuracy are highly prompt-dependent... minor prompt changes can cause large changes in the distribution of generated labels."
  4. OpenAI's official position: "Start with the smallest prompt that preserves the product contract, then tune reasoning effort, verbosity, tool descriptions, and output format." and "Keep each section short. Add detail only where it changes behavior."
  5. DeepSeek's official position: "It is recommended to keep system instructions within a clear and concise range, avoiding excessive length that occupies the context window."

7.2 The Actual Token Efficiency Principle

The verified principle is not a specific token count but a structural approach: discrete, labeled, verifiable constraints outperform long prose paragraphs because each constraint is independently checkable by the model. This is consistent with OpenAI's trigger/instruction pair approach, Anthropic's XML-tagged section approach, and Gemini's prefix-based demarcation approach.

PART 8: THE COMPLETE CROSS-PLATFORM SKILL TEMPLATE

Below is the smallest contract that passes validation on all four platforms, constructed exclusively from verified structural primitives. It uses Markdown headings (works on all platforms) as the structural delimiter, with content structured to map onto each platform's specific requirements.

# Role
[1-2 sentences: function, context, job. Placed first per OpenAI's 7-part structure]

# Personality
[Tone, demeanor, collaboration style. Per OpenAI GPT-5.5 guidance.
Claude also uses this for behavioral anchoring.]

# Task
[User-visible outcome. Per OpenAI: "Goal." Per Claude: "Be clear and direct."
Per Gemini: system_instruction defines task framework.
Per DeepSeek: system prompt defines behavioral rules.]

# Constraints
[Discrete, verifiable rules. One per line. Per OpenAI: Constraints section.
Per Claude: explicit instructions. Per DeepSeek: system prompt rules.
Per Gemini: constraint adherence via system_instruction.]

# Output Format
[Precise format specification. Include JSON example if JSON mode is needed.
Per DeepSeek: MUST contain "json" keyword and example for JSON mode.]

# Success Criteria
[What must be true before the final answer. Per OpenAI GPT-5.5 structure.]

# Stop Rules
[When to retry, fallback, abstain, ask, or stop. Per OpenAI GPT-5.5 structure.
Claude: equivalent to contingency/error handling.]

# Examples
[3-5 input-output pairs. Per Anthropic: "Examples are one of the most
reliable ways to steer Claude's output format, tone, and structure."
Per OpenAI: trigger/instruction pairs with few-shot examples.
Per Gemini: few-shot prompting establishes patterns.]
